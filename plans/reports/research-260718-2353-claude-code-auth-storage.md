# Research: Claude Code Auth Storage & Account-Switching Mechanisms (macOS)

Scope: how `claude` (Anthropic CLI) stores credentials on macOS, what defines a "profile," every viable switching mechanism, prior art, and risks — for VibeProxy (native macOS menubar app: quick profile switch + usage display + auto-switch on quota exhaustion).

## 1. Credential storage on macOS

| Location | Purpose | Verified |
|---|---|---|
| macOS Keychain, service `Claude Code-credentials`, account = `$USER` | Primary OAuth token store on macOS | Confirmed: local machine has **no** `~/.claude/.credentials.json` (macOS uses Keychain exclusively, unlike Linux/Windows). Corroborated by [official docs](https://code.claude.com/docs/en/authentication) and multiple GitHub issues. |
| `~/.claude/.credentials.json` (mode 0600) | Linux/Windows credential store; macOS fallback if the file exists it's read *instead of* Keychain | Official docs + [issue #29816](https://github.com/anthropics/claude-code/issues/29816) (SSH/headless workaround: export the Keychain item to that file via the `security` CLI) |
| `~/.claude.json` key `oauthAccount` | Account/org metadata, NOT the token itself | Verified directly on this machine: `oauthAccount` block contains `accountUuid, organizationUuid, organizationName, emailAddress, displayName, billingType, seatTier, userRateLimitTier, organizationRateLimitTier, organizationRole, organizationType, workspaceRole, subscriptionCreatedAt, profileFetchedAt, hasExtraUsageEnabled, claudeCodeTrialEndsAt`. Top-level `hasAvailableSubscription: bool` also lives in `~/.claude.json`. |
| `~/.claude/settings.json` (user) / `.claude/settings.json` (project, git-tracked) / `.claude/settings.local.json` (project, not tracked) / managed `managed-settings.json` | Non-secret config, incl. `apiKeyHelper`, `forceLoginMethod`, `forceLoginOrgUUID`, `env` block | Confirmed on this machine (`~/.claude/settings.json` has no secrets, just UI prefs). Precedence: Managed > Local(project) > Project > User (per [settings docs](https://code.claude.com/docs/en/settings)). |
| `CLAUDE_CONFIG_DIR` env var | Redirects **everything** — settings, `.credentials.json`/Keychain, history, plugins, project registry — to an alternate directory | Undocumented officially but real; requested in [issue #33430](https://github.com/anthropics/claude-code/issues/33430) (closed "not planned" — Anthropic won't formally document it, but doesn't dispute it works). |

### Keychain item internal shape (from community teardown, not from this machine)
```json
{"claudeAiOauth":{
  "accessToken": "<redacted; token prefix 'oat01', ~24h expiry>",
  "refreshToken": "<redacted; token prefix 'ort01'>",
  "expiresAt": 1234567890000,
  "scopes": ["user:inference","user:profile","user:sessions:claude_code", "..."],
  "subscriptionType": "max-or-pro",
  "rateLimitTier": "..."
}}
```
Source: multiple GitHub issue reproductions (#21765, #34306, #43639, #68398) and a binary-analysis gist. **Not independently re-verified on this machine** (no local keychain read was performed — blocked by sandbox policy for this session; treat field names as high-confidence but unverified-by-me).

### Auth method precedence (official, from [authentication docs](https://code.claude.com/docs/en/authentication))
1. Cloud provider env vars (`CLAUDE_CODE_USE_BEDROCK` / `_VERTEX` / `_FOUNDRY`)
2. `ANTHROPIC_AUTH_TOKEN` (Bearer header)
3. `ANTHROPIC_API_KEY` (X-Api-Key header) — prompts once to approve in interactive mode
4. `apiKeyHelper` script output
5. `CLAUDE_CODE_OAUTH_TOKEN` (long-lived token from `claude setup-token`, 1yr, model-requests only)
6. Subscription OAuth from `/login` (Keychain/`.credentials.json`) — default for Pro/Max/Team/Enterprise

**Critical for VibeProxy:** `apiKeyHelper` and `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` **do not apply** to OAuth subscription sessions — the docs explicitly say helper/env vars are for the API-key path only, and Claude Desktop/OAuth sessions never call `apiKeyHelper`. This kills `apiKeyHelper` as a switching mechanism for Pro/Max multi-account use; it's only relevant if profiles use Console API keys instead of subscription login.

## 2. What defines a "profile" (full session state)

To switch accounts cleanly with **zero re-login**, you must swap, atomically, as a unit:
1. Keychain item `Claude Code-credentials` (or the config-dir's `.credentials.json`) — `accessToken`, `refreshToken`, `expiresAt`, `scopes`, `subscriptionType`, `rateLimitTier`.
2. The `oauthAccount` block in `~/.claude.json` (or the config-dir's own `.claude.json`) — account/org identity metadata Claude Code displays and uses for `/status`, `forceLoginOrgUUID` checks, etc.

Both must correspond to the *same* account — mismatched Keychain-token vs. `oauthAccount` metadata is how tools like CCSwitcher describe their atomic-swap requirement. Everything else (settings, history, projects registry, plugins) is *not* part of the account identity and can be shared or per-profile depending on design choice.

## 3. Switching mechanisms — enumerated with restart requirement

| Mechanism | How | Needs restart of running `claude` session? | Notes |
|---|---|---|---|
| **`CLAUDE_CONFIG_DIR` per profile** | Env var set before `claude` launches; each dir has isolated Keychain entry (macOS: path is SHA-256 hashed into a distinct Keychain service name — claimed by community sources, not independently verified) or `.credentials.json` | Yes — read once at process start; existing terminal sessions are pinned to whatever dir was set at launch | **Official-adjacent** mechanism (undocumented but Anthropic closed the "document it" request as not-planned rather than "this is unsupported/disallowed"). Cleanest isolation: each profile does its own real `/login`, so no token extraction/reuse — lowest ToS risk (see §5). Does NOT enable switching an already-running session; only affects *new* `claude` invocations. |
| **Keychain item swap** (via `security` CLI, replacing the single global `Claude Code-credentials` entry) + overwrite `oauthAccount` in `~/.claude.json` | External tool atomically replaces the single global Keychain entry + JSON block | Ambiguous/uncertain — community reports (`claude-swap`) say macOS Keychain reads are cached ~30s by Claude Code, so a running session *may* pick up new creds without a hard restart once cache expires, but this is not guaranteed and is described as unreliable enough that tools recommend restarting anyway | This is what **CCSwitcher** and **ccswitch (vyshnavsdeepak)** do. Single shared `~/.claude` — no per-profile settings/history isolation. Requires `security` CLI calls, which can trigger macOS Keychain access prompts unless the app's code-signing identity is already trusted for that keychain item (ACL). |
| **Swap/symlink `~/.claude/.credentials.json`** | Only relevant if the *file* fallback is in use (macOS normally uses Keychain, but Claude Code prefers the file if it exists) | Same as above — implementation detail, functionally same swap as Keychain but simpler I/O, no `security` CLI/prompts | Good escape hatch: **force Claude Code onto file-based creds even on macOS** by creating the file once, then all subsequent profile swaps are plain file writes with no Keychain prompts at all. Tradeoff: credentials sit on disk unencrypted (mode 0600) instead of in Keychain. |
| **`apiKeyHelper`** | Settings-defined script returning a key; re-invoked every 5 min or on 401 (`CLAUDE_CODE_API_KEY_HELPER_TTL_MS` configurable) | **No** — this is the one mechanism that updates a *running* session without restart | Only works for API-key/bearer-token auth paths, **not** for OAuth subscription (Pro/Max) sessions per official docs. Relevant only if VibeProxy manages Console API-key-based profiles, not `.ai` login profiles. |
| **`ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` env injection** | Set before launch | Yes (env read at startup; also gated by an interactive one-time approval prompt) | Same limitation as above — API-key path only, and overrides subscription login if both present (can cause confusing auth failures if stale). |

**Bottom line:** nothing lets you hot-swap an OAuth/subscription account inside an already-running `claude` process. Every viable mechanism for Pro/Max profiles requires the *next* `claude` invocation to pick up new state — either via a new `CLAUDE_CONFIG_DIR`-scoped process, or via Keychain/file swap before the next launch (with a ~30s uncertainty window if a session is already live and macOS-Keychain-backed).

## 5. Constraints / risks

1. **Anthropic ToS — OAuth token reuse outside official client (HIGH RISK, recently tightened).** As of ~Feb 2026, Anthropic's [Claude Code legal/compliance docs](https://code.claude.com/docs/en/legal-and-compliance) state that using OAuth tokens obtained through Claude Free, Pro, or Max accounts in any other product, tool, or service — including Anthropic's own Agent SDK — is not permitted and constitutes a Consumer ToS violation. Server-side enforcement (Jan 2026) now returns an explicit "credential only authorized for Claude Code/claude.ai" error for non-Claude-Code/claude.ai callers. **Implication for VibeProxy: never make direct Anthropic API calls using an extracted OAuth token (e.g., to poll usage/quota).** Only (a) orchestrate the real `claude` binary (spawn/env-swap/launch), or (b) read locally-generated data (Keychain-stored token used only to feed back to `claude` itself, JSONL transcripts, statusline output). This is confirmed as the accepted-vs-banned line in a community writeup ("[Two Multi-Account Claude Code Architectures](https://dev.to/vainamoinen/two-multi-account-claude-code-architectures-one-anthropic-accepts-one-they-ban-2om7)"): per-profile `CLAUDE_CONFIG_DIR` + real per-profile `/login` = accepted pattern (Anthropic closed the docs-request issue as "not planned," i.e., tacit tolerance, not endorsement); relay-server pooling many accounts' tokens on a non-official-client server = actively banned/enforced (cites the "OpenClaw ban," April 2026-era pattern). CCSwitcher's design choice to refresh via `claude auth status` subprocess rather than hitting the refresh endpoint directly is explicit precedent for staying on the safe side of this line.
2. **Multiple accounts per se are not banned.** An Anthropic Claude Code team member is quoted saying holding multiple Max accounts isn't a ToS violation; enforcement targets reselling/pooling, not individual multi-account switching. Account *sharing* (one account, multiple people) is separately prohibited under the acceptable-use policy — irrelevant to VibeProxy's single-user use case but worth noting if the feature set ever expands to team pooling.
3. **Keychain access prompts / macOS app sandbox.** Reading/writing another process's Keychain item from a sandboxed app normally triggers a system password prompt unless: the item's ACL already trusts the requesting app's code-signing identity, or the app is unsandboxed and uses the `security` CLI as a subprocess (CCSwitcher's approach — reads via `/usr/bin/security find-generic-password` specifically to avoid repeated GUI prompts). A hardened/sandboxed VibeProxy with the `keychain-access-groups` entitlement would need the *same team ID / access group* as whatever wrote the "Claude Code-credentials" item — since `claude` itself isn't a VibeProxy-signed binary, standard Keychain Access Groups won't apply; VibeProxy will most likely need the (non-App-Store, notarized, unsandboxed) `security`-CLI-subprocess pattern, or use the file-based-credentials escape hatch (§3) to sidestep Keychain entitlements entirely.
4. **Token refresh races.** Access tokens expire roughly 24h from login; refresh token is used opportunistically. Multiple open GitHub issues (#21765, #34306, #42904, #68398, #43639) describe refresh failures: refresh token not honored after copying creds to a new machine, no auto-refresh on process restart forcing daily re-login, and `subscriptionType`/`rateLimitTier` being cached indefinitely (stale after plan upgrades). **Implication:** VibeProxy swapping Keychain items externally could itself trigger or mask refresh races — swap operations should be atomic (write new before invalidating old) and the app should treat `claude`'s own `/status`/`auth status` output as ground truth rather than trusting the cached `oauthAccount` blob indefinitely.
5. **No authoritative live-quota API for third parties.** Because direct OAuth-token API calls are banned (risk #1) and there's no public "check my quota" endpoint sanctioned for third-party tools, VibeProxy's usage display is necessarily either (a) historical/local (JSONL-based cost estimate — always somewhat stale, no true "remaining quota"), or (b) live but requires spawning `claude` itself (e.g., headless `-p` prompt or reading its statusline output) per profile to surface Anthropic's real rate-limit headers — adds latency/cost (a real inference call) just to check usage. This is an open design tension: auto-switch-on-quota-exhaustion needs live signal, but the only ToS-safe live signal comes from the CLI itself.
6. **`CLAUDE_CONFIG_DIR` is explicitly undocumented-by-choice** (Anthropic closed the "please document it" issue as not-planned). It works today across multiple community tools but carries higher break-with-a-future-update risk than officially documented settings, since Anthropic isn't committing to preserving it.

## Recommendation (ranked)

1. **Primary mechanism — per-profile `CLAUDE_CONFIG_DIR`, real `/login` per profile.** Lowest ToS risk (Architecture-B pattern, tacitly accepted), no Keychain-entitlement fight, clean isolation (settings/history/plugins optionally shared via symlink-back for non-secret files). Downside: switching = spawning a new `claude` process with the right env var, not live-patching a running session — acceptable since no mechanism achieves true hot-swap anyway (§3).
2. **Secondary mechanism — file-based credentials override.** For the "currently active/default" profile that a bare `claude` invocation (no `CLAUDE_CONFIG_DIR`) should use, force file-based creds (`~/.claude/.credentials.json`) instead of Keychain by creating the file once via the documented SSH-workaround command. This sidesteps macOS Keychain entitlement/ACL problems for the default profile entirely, at the cost of unencrypted-on-disk tokens (mode 0600, same trust boundary as the rest of `~/.claude`).
3. **Usage display — local JSONL parsing (ccusage-style) for historical cost, plus statusline/`/usage` for live quota when a profile's `claude` is actively running.** Do not call the Anthropic API directly with extracted OAuth tokens for usage polling — this is the one hard ToS line (risk #1) and it's actively enforced server-side.
4. **Auto-switch trigger — reactive, not polling.** Detect rate-limit exhaustion from the running `claude` process's own output/exit behavior (429/"rate limit" text, or statusline percentage crossing threshold while that profile is active) rather than building a separate poller that hits Anthropic's API per-account — avoids both the ToS risk and the added inference cost of check-in calls.

Architectural fit: Swift/SwiftUI native menubar app (matching CCSwitcher's proven stack) is the right call for VibeProxy given "native macOS app" is a stated requirement — Tauri/Electron (cc-switch's stack) adds a runtime and doesn't buy anything for this scope, and Swift gives direct, prompt-free `security` CLI subprocess access plus first-class menubar (`NSStatusItem`) support.

## Unresolved questions

- Whether `CLAUDE_CONFIG_DIR` truly generates a per-path SHA-256-hashed Keychain service name on macOS (claimed by a secondary blog source, not verified against Claude Code's binary or an official source) — worth a direct empirical check (set two different `CLAUDE_CONFIG_DIR` values, dump Keychain, compare service names) before relying on it architecturally.
- Exact behavior of a *running* `claude` session when its backing Keychain item is swapped externally: does it re-read on next request, only after the ~30s cache window, or not until process restart? Community sources conflict/hedge (claude-swap says "optional restart... for instant," implying eventual pickup without restart, but this isn't confirmed against source or changelog).
- How `claude-swap`'s "adaptive polling" for auto-switch actually sources live usage numbers — README excerpt doesn't confirm whether it's local-JSONL-based, statusline-based, or a direct API call (which would conflict with the Feb 2026 OAuth-token policy). Worth a closer code read before treating it as a safe precedent.
- Whether Anthropic's Feb 2026 enforcement scopes to "any Anthropic API call not made by the literal `claude`/`claude.ai` client binary" or has a narrower technical fingerprint (e.g., User-Agent based) that a wrapper spawning `claude` as a real subprocess would trivially satisfy either way. Current sources describe outcome/policy, not enforcement mechanics.
- No independent verification was performed of this machine's actual Keychain item (`security` calls were blocked by this session's sandbox policy) — all Keychain-content claims here are sourced from public GitHub issues/gists, not directly confirmed locally. `~/.claude.json` structure (`oauthAccount` field list) **was** verified directly on this machine.

Status: DONE
Summary: Documented macOS credential storage (Keychain "Claude Code-credentials" + `~/.claude.json` `oauthAccount`), the two things that must swap atomically to switch accounts, all switching mechanisms ranked by restart-requirement, 7+ prior-art tools with mechanisms, and a hard ToS constraint (Feb 2026 ban on using OAuth tokens outside the official client) that should drive VibeProxy toward CLAUDE_CONFIG_DIR-per-profile + local-log-based usage display rather than direct-API polling.
Concerns: Two load-bearing technical claims (CLAUDE_CONFIG_DIR's macOS Keychain-hashing behavior, and whether a running session picks up a swapped Keychain item without restart) come from secondary/community sources only and should be empirically verified (via `security` CLI testing) before the plan finalizes the switching architecture.
