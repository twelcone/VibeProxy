# QA Report — `vibeproxy` CLI

Date: 2026-07-22 · Branch: `refactor/core-extraction`

Adversarial sweep of every CLI command before any UI work. All runs isolated via `VIBEPROXY_DIR`
(and a temp `HOME` for install), so real state was untouched — verified clean after.

**45 manual checks, 45 pass** (+ 6 hermetic integration tests in `crates/vibeproxy-cli/tests`).
No CLI defect found. Two earlier "failures" were harness bugs (used `main` for the label `Main`);
re-run with the real label/id, all correct.

## Notable behaviours confirmed

- **Switch → shell → claude loop works for real**: `switch <p>` redirects `CLAUDE_CONFIG_DIR`;
  `claude auth status` follows it; the default account correctly UNSETS the variable.
- **remove-active re-points** to the first remaining profile (and its active-path); remove-last
  clears active-path.
- **Range edges**: a future range yields `Total: 0 tokens`; a reversed range does not crash.
- **Tilde**: a literal `~/` argument is expanded.
- **Secrets**: no token appears in any stdout/stderr across all commands.

## Real bug found + fixed during testing (committed)

The shell snippet hardcoded `~/.vibeproxy/active-path` while the CLI/app write to
`$VIBEPROXY_DIR/active-path` when set — so a relocated state dir silently broke switching. Snippet now
resolves `${VIBEPROXY_DIR:-$HOME/.vibeproxy}`; frontend renders the backend snippet (no drift).

## Minor UX observations (not bugs — your call)

1. **Label matching is case-sensitive.** `switch Main` works; `switch main` errors. A user typing the
   label as shown in `list` is fine, but case-insensitive matching would be friendlier. Risk:
   ambiguity if two labels differ only by case. Recommendation: leave exact for now.
2. **`list` (human) shows labels, not ids.** Switching by id needs `list --json`. Consistent with
   "labels for humans, ids for scripts", but the id isn't discoverable in the plain view. Could add a
   dim id column. Low priority.

## Not covered (honest gaps)

- **Linux/WSL execution.** The `FileStore` credential backend (phase 5 Track A) is unit-tested on
  macOS but its wiring as the non-macOS default is only CI-verifiable on a Linux runner.
- **CLI ↔ app concurrent writes.** Both write `config.json`/`active-path`; last-writer-wins is
  expected but not stress-tested here.
- **auto over-threshold path.** `auto` was exercised only with an under-threshold live account
  ("nothing to do"); the switch/blocked branches are covered by core unit tests, not end-to-end here.
